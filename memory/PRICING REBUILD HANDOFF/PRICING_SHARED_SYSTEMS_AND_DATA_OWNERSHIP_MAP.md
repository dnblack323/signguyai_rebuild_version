# PRICING SHARED SYSTEMS AND DATA OWNERSHIP MAP

**Status:** Documentation/investigation only. No code changed, no migrations created.

**Purpose:** For every pricing-relevant entity, name exactly one owning system in the rebuild, who is allowed to read/write it, what must never be duplicated, and whether it follows the "live" or "frozen snapshot" rule (see `PRICING_ISSUES_INVENTORY_AND_REMEDIATION_PLAN.md` §4E for the full live-vs-snapshot architecture statement).

---

## Cross-cutting ownership rule (applies to every row above)

**One collection, one shape, one set of consumers.** Any time a second system (Wrap Lab, Webstores, a legacy `jobs`/`job_items` path) needs the same kind of data a shared entity already represents, the rebuild rule is to make that second system a *consumer* of the shared entity — never a second producer with its own shape. This is the single structural fix underneath nearly every "Confirmed" problem in this document and in `PRICING_MODULE_EXTRACTION_INVENTORY.md`.
