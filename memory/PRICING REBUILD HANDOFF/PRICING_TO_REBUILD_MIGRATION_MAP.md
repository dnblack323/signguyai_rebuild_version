## Migration principles that apply across every row above

1. **Never rewrite a price that already happened.** A committed `PricingSnapshot` (or its legacy equivalent) for a job that already shipped is a historical fact, not a live calculation — migration transforms its *shape* (float dollars → cents, field renames) but never its *value*.
2. **Config migrates; unexplained history is not fabricated.** Where the legacy system never captured a "why" (e.g., a manual override with no reason field), the migration synthesizes a clearly-labeled placeholder (`reason: "other"`, a note that provenance predates the rebuild) rather than inventing a false explanation.
3. **Curated over automatic where trust is uncertain.** Webstore product re-categorization and Wrap Lab material promotion are explicitly Owner-reviewed, not blind automatic migrations, because the legacy data's accuracy is already in question (that's WHY it's being rebuilt).
4. **"Needs More Investigation" rows are not blocking Phase 2's start** — they are read-before-you-build items for whichever build-order phase actually touches that specific category (Apparel/Services/Promotional details block Phase 5, not Phase 2).
