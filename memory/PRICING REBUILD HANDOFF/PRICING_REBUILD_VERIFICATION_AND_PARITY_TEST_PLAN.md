# PRICING REBUILD VERIFICATION AND PARITY TEST PLAN

**Status:** Documentation only. No tests implemented beyond Sprint 1's already-passing 58 unit tests (Money, permissions, change log, golden-formula framework). This plan defines what every future phase must test before being considered done.

---

## How to read this document

For each rebuild area: what must be preserved exactly from the legacy behavior (parity), what is intentionally changed (and why), the required test cases, and the acceptance criteria that close that area out. "Preserve From Legacy" does not mean "preserve the legacy code" — it means the *business-visible outcome* (a specific price for a specific input) must match, even though the implementation underneath is completely rewritten.

| **Apparel examples** | Legacy `calculate_apparel()` output for representative brand/style/quantity-tier/placement/decoration-method combinations | Float → cents; standardized snapshot output | Golden-formula suite per the Phase 3 pattern | New engine output == legacy output for every golden case |

| **Manual override tests** | The concept of a manual override (users have always been able to type a different price) | Adds `PricingOverride` provenance (reason, approver, permission) that did not exist before (M-15) | Attempt an override as Staff (should route to pending approval, cannot finalize below cost, per `Permission.PRICING_OVERRIDE` mapping — already unit-tested in Sprint 1's `test_pricing_permissions_sprint1.py`); attempt as Admin/Owner (self-approving) | An override always produces a populated `PricingOverride` block; a Staff-initiated below-cost override never silently finalizes |
| **Margin warning tests** | The concept that a thin-margin sale should be flagged | Warnings become typed with severity, replacing free-text strings (M-13/M-24) | Golden-formula case with a price above cost but below the target margin threshold triggers exactly one `margin_warning` (already proven in Sprint 1's `test_golden_formula_framework.py::test_margin_warning_output`) | Every category's golden-formula suite includes at least one margin-warning case |
| **Below-cost approval tests** | The concept that below-cost sales are unusual and worth flagging | Below-cost commits require `PRICING_OVERRIDE` + a reason, where today they require nothing (M-14) | Golden-formula case with an override price below total cost triggers exactly one `below_cost` critical warning (already proven, `test_golden_formula_framework.py::test_below_cost_warning_output_and_manual_override`); a live-endpoint integration test (Phase 6) that a Staff user's below-cost attempt is actually rejected/queued, not just flagged | The permission is enforced at the API layer, not merely represented in a schema |
| **Snapshot freeze tests** | Nothing to preserve from legacy here — this is a NEW guarantee (today's snapshots CAN be silently invalidated by later config changes, since they aren't consistently isolated from live config in every surface, e.g., Wrap Lab's compute-and-save-together pattern) | A committed snapshot is provably immutable against later Foundation/config changes | Commit a `PricingSnapshot`, then change the relevant material cost/labor rate/markup, then re-fetch the original snapshot and confirm its stored price is unchanged | A snapshot's `final_price_cents` never changes after `snapshotted_at`, regardless of any later config write |
| **Material-cost change tests** | N/A (new guarantee, same as above) | A material cost change affects only FUTURE calculations, never past snapshots | Change a material's `cost_per_sqft`, immediately re-run the SAME calculator inputs live (should reflect the new cost), then check a previously-committed snapshot using that material (should NOT reflect the new cost) | Live calculation reflects new config immediately; historical snapshots do not |
| **Quote/Order integration tests** | The general shape of "a Quote can become an Order, an Order can become an Invoice" (existing conversion functions `generate_quote_from_order`/`generate_invoice_from_order`, confirmed to exist and work today) | The underlying snapshot each conversion step reads comes from ONE standardized `PricingSnapshot` instead of `job_tickets.pricing_snapshot`'s current ad hoc shape, and no longer falls back to the legacy `jobs`/`job_items` sync path (confirmed current anti-pattern in `routes/invoices.py`) | End-to-end: price an item live, commit, convert Quote → Order → Invoice, confirm the exact same price (to the cent) appears at every stage | Zero price drift across the Quote → Order → Invoice chain; zero calls to `sync_job_items_from_embedded_line_items`-equivalent legacy paths |
| **Wrap Lab parity tests** | See "Vehicle Wrap examples" above — same documented-comparison-then-Owner-approval principle, not blind parity | Wrap Lab's compute-and-save-together pattern (`routes/wrap/core.py` L1254-1289) becomes the same live-then-commit two-step pattern every other category has | A wrap ticket's pricing PUT endpoint no longer silently persists a snapshot on every recalculation — a distinct "commit" action is required | Wrap Lab UI has a visibly distinct "still live/editing" state versus a "committed" state, matching every other category |
| **Webstore pricing tests** | The concept of a public retail price and a fundraiser profit split (both confirmed to work today, `compute_event_profit_allocation()`) | The internal cost/price behind that public price now comes from a real calculator instead of a manually-typed `base_cost` (M-02) | Create a Webstore product mapped to a real category, confirm its internal `PricingResult` is non-null and matches what the standalone calculator would produce for the same inputs | Zero Webstore products with a calculator-disconnected price after Phase 8 |
| **Webstore override tests** | The concept of a store owner being able to set a different public price than the internally suggested one (Assumption to Verify whether this exists today at all — Needs More Investigation #7) | The override, if it exists today with no provenance, gains the same `PricingOverride` treatment every other category's override gets, and — critically — per the final rebuild rules, it must never erase the internal calculated cost, margin, price provenance, owner share, platform fee, or frozen snapshot | A public override change never deletes or nulls the internal cost/margin fields; a below-internal-cost public override triggers the standard `below_cost` warning | All 5 protected fields (internal cost, margin, provenance, owner share, platform fee) remain populated after any public override edit |
| **Fundraiser/event pricing tests** | `compute_event_profit_allocation()`'s percentage/fixed-per-item modes and running-total tracking (confirmed sound, `routes/webstores.py` L1034-1073) | Its INPUT (`total_profit`) becomes more accurate once product pricing is fixed — the allocation math itself is unchanged | Re-run the existing profit-allocation test cases (if any exist; if not, this phase adds them) against both the legacy and rebuilt product-pricing paths, confirming the allocation formula itself produces identical output for identical `total_profit` inputs | Allocation math output is provably unchanged by the pricing-engine swap; only the accuracy of its input improves |
| **Profitability report tests** | The general concept of a margin/profit dashboard (exists today, `routes/profit_analytics.py`) | Its data source moves from the confirmed-legacy `jobs`/`job_items` collections to `PricingSnapshot` records exclusively (M-18/M-19) | Price and commit a test order, confirm it appears correctly in a profitability report within the same test run with no manual "sync" step; confirm a report generated BEFORE this phase (reading legacy data) and one generated AFTER (reading snapshots) agree on historical totals for the same historical orders (a true parity check on old data) | New reporting reads `PricingSnapshot` exclusively; historical totals for pre-rebuild orders are unchanged versus the legacy report |
| **Inventory/vendor cost tests** | `services/inventory_service.py`'s reservation/allocation logic (confirmed sound, no changes recommended) | Only the potential new link from vendor cost data into `pricing_materials.cost_per_sqft` (Needs More Investigation) | If a link is built: a PO receipt at a different cost produces a visible, actionable signal (not silent) | Inventory reservation/allocation behavior is completely unchanged; only a new, additive cost-update signal is tested |
| **Permission tests** | Nothing (no dedicated pricing permission existed before Sprint 1) | New `PRICING_VIEW`/`PRICING_EDIT_FOUNDATION`/`PRICING_OVERRIDE` permissions, already unit-tested (Sprint 1, `test_pricing_permissions_sprint1.py`, 15+ passing assertions including explicit no-regression checks against every pre-existing role permission) | Live-endpoint integration tests (Phase 6) that every pricing-config-save and override-commit route actually enforces the permission, not just the schema | Every pricing-mutating endpoint has a live test confirming a user without the permission is rejected (403), and a user with it succeeds |
| **Change-log tests** | Nothing (no audit trail existed before Sprint 1) | New `PricingChangeLog`, already unit-tested (Sprint 1, `test_pricing_change_log.py`, using an in-memory fake collection) | Live-endpoint integration tests (Phase 2/6) that every settings/material/category-config save route produces exactly one new `PricingChangeLog` entry with the correct old/new value | Every pricing-config-mutating route under real MongoDB produces a verifiable log entry; a code-review grep check confirms no such route is missing the call |
